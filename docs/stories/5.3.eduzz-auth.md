## Story

**As a** recipient do tracking stack que usa Eduzz,  
**I want** a estratégia de autenticação do webhook Eduzz documentada e implementada,  
**so that** saiba exatamente como proteger o endpoint de webhooks falsos.

## Acceptance Criteria

1. `docs/platforms/eduzz.md` atualizado com seção de hardening descrevendo as opções de autenticação disponíveis na Eduzz
2. Se Eduzz oferece token/assinatura verificável: implementado em `functions/webhook/eduzz/[slug].js` com `timingSafeEqual`
3. Se Eduzz usa apenas IP allowlist: documentado com a lista de IPs conhecidos e comentário explicando a limitação (não implementável no Cloudflare Worker sem Workers for Platforms)
4. Qualquer nova env var necessária adicionada em `wrangler.toml.example`
5. Comentário no adapter explica a estratégia escolhida e por quê

## Dev Notes

**Estratégia pragmática:** se Eduzz não oferece assinatura verificável, a defesa em profundidade é:
1. Slug UUID obscuro (já implementado)
2. Validação do shape do payload (já feita no parse)
3. Documentação dos IPs Eduzz para allowlist manual (informativo)

**`timingSafeEqual` já existe em `functions/webhook/_utils.js`** — usar se houver token verificável.

**Regra do projeto:** adapters retornam 404 para slug errado (não 401) — indistinguível de rota inexistente. Manter esse comportamento.

## QA Results

### Review Date: 2026-04-25
### Reviewer: @qa (Quinn)
### Decision: PASS ✅

**Análise de Segurança:**
- Implementada verificação HMAC-SHA256 via header `x-signature`.
- A estratégia foi refinada para usar o mesmo helper genérico `verifyHmacSignature` introduzido na story da Kiwify, garantindo consistência.

**Rastreabilidade:**
- A pesquisa sobre o mecanismo de assinatura da Eduzz confirmou o uso de HMAC-SHA256, substituindo a estratégia inicial baseada apenas em IP allowlist por algo muito mais robusto.

**Observações:**
- A transição do adapter Eduzz para suporte a HMAC eleva o padrão de segurança de todo o stack, visto que ele serve como base para novos adapters.
