## Story

**As a** recipient do tracking stack,  
**I want** o adapter Hotmart verificando o header `X-Hotmart-Hottok`,  
**so that** webhooks falsos sejam rejeitados antes de processar qualquer compra.

## Acceptance Criteria

1. `functions/webhook/hotmart/[slug].js` atualizado para verificar `X-Hotmart-Hottok` header usando `timingSafeEqual` de `functions/webhook/_utils.js`
2. Request sem o header ou com valor errado retorna `401 Unauthorized`
3. `HOTMART_HOTTOK` adicionado em `wrangler.toml.example` como env var obrigatória quando usar Hotmart
4. `functions/webhook/_utils.js` atualizado com helper `verifyHotmartHottok(request, env)` — ou verificação inline se preferível (decisão do dev)
5. `docs/platforms/hotmart.md` atualizado com instruções de onde configurar o hottok no painel Hotmart e como setar `HOTMART_HOTTOK`
6. Webhook continua funcionando normalmente quando `HOTMART_HOTTOK` está corretamente configurado

## Dev Notes

**`timingSafeEqual` já existe em `functions/webhook/_utils.js`** — não reimplementar. Verificar assinatura exata antes de usar.

**Comportamento quando `HOTMART_HOTTOK` não está configurado:** deve retornar 500 (env var obrigatória ausente) — consistente com o padrão do slug. Não deve silenciosamente aceitar qualquer hottok.

**Header Hotmart:** `X-Hotmart-Hottok` — valor configurado pelo recipient no painel Hotmart → Configurações → Webhooks.

**Regra de segurança:** NUNCA usar comparação `===` para verificar tokens — timing attack. Sempre `timingSafeEqual`.

**Arquivos afetados:**
- `functions/webhook/hotmart/[slug].js`
- `functions/webhook/_utils.js` (possível)
- `wrangler.toml.example`
